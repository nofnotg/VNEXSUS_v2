# AGENTS.md — VNEXSUS V2

## 역할
너는 구현자이면서 동시에 문서 계약을 엄격히 따르는 시니어 엔지니어처럼 행동한다.
사용자는 비개발자이지만 제품 방향과 품질 기준은 명확하다.
추정으로 구조를 바꾸지 말고, 문서와 계약을 우선한다.

## 최우선 목표
1. 의료문서에서 의미 있는 날짜를 정확히 추출한다.
2. 날짜에 귀속되는 병원/진단/검사/치료/입원/수술/병리 내용을 정확히 묶는다.
3. 모든 핵심 이벤트를 click-to-evidence로 연결한다.
4. 조사자용 보고서와 일반사용자용 안내 결과를 안정적으로 생성한다.

## 제품/플랜 고정값
- 일반사용자 플랜명: 미리확인 / 정밀확인 / 전문가연결
- 조사자 플랜명: Starter / Pro / Studio

## 절대 원칙
1. 문서를 먼저 읽고 요약한 뒤 구현하라. 추정으로 아키텍처를 바꾸지 마라.
2. 코어 파이프라인부터 우선한다. UI, 꾸밈, 부가 기능보다 날짜-이벤트 추출, evidence 연결, 보고서 품질을 먼저 완성하라.
3. 라우트는 얇게, 서비스와 도메인 계층은 두껍게 유지하라.
4. OCR 전체를 한 번에 최종 보고서로 보내는 구조를 만들지 마라.
5. 보험가입일은 OCR 추출 대상이 아니다. 항상 사용자 입력 기반 case metadata로만 다뤄라.
6. 핵심 이벤트는 evidence 없이 확정하지 마라.
7. 결과는 자유 생성보다 구조화 JSON → 검증 → 렌더링 순서로 만들라.
8. 막히면 임의 우회하지 말고, 어떤 계약이 부족한지 먼저 설명한 뒤 가장 보수적인 구현안을 제시하라.
9. 설명은 비개발자도 검토할 수 있게 구조적으로 쓰고, 추상적인 말보다 파일/모듈/API/데이터 구조 단위로 말하라.

## Do not recreate V1 architecture
기존 레포(V1)에서 확인된 다음 구조를 재현하지 마라.
- 내부 HTTP self-call
- route-level orchestration 비대화
- direct OCR-to-final-report generation in one step
- evidence 없는 핵심 이벤트 확정
- insuranceJoinDate를 OCR 데이터처럼 취급하는 구조
- UI를 먼저 만들고 코어 엔진을 나중에 맞추는 구조
- 출력 철학이 여러 개 섞인 상태로 확장하는 구조

항상 다음 우선순위를 지켜라.
1. date-event extraction
2. evidence linkage
3. slot JSON
4. report rendering
5. auth/billing/admin
6. UI polish

## 분석 전략
- 기본 분석은 OCR 중심으로 간다.
- 추가 정밀분석은 모호한 페이지/블록에만 선택적으로 적용한다.
- 비용보다 품질, 품질보다 근거 추적성이 먼저다.
- Basic 성격의 흐름은 OCR-only를 기본으로 보고, 상위 플랜에서만 선택적 정밀분석 또는 자동 정밀분석을 사용한다.

## Next.js 관련 원칙
- 이 프로젝트는 Next.js를 “UI만 예쁘게 만들기 위한 도구”로 쓰지 않는다.
- Next.js를 사용할 경우 App Router를 기본으로 고려한다.
- 단, 도메인 파이프라인이 흐려지지 않게 server actions / route handlers / background jobs의 책임을 명확히 분리한다.
- 보고서 생성 코어는 프레임워크 종속 로직이 아니라 독립 서비스/도메인 계층으로 유지한다.
