# PRD — 비정형 의료문서 대응 엔진 v1
_작성 목적: 아이디어/철학 문서를 제품 요구사항 문서로 전환_

## 1. 문서 개요

### 1.1 제품명(가칭)
비정형 의료문서 대응 엔진  
(Unstructured Medical Document Response Engine)

### 1.2 문서 목적
이 문서는 비정형 의료문서를 단순 OCR 결과가 아니라 **검증 가능한 시간축 정보**로 변환하는 제품의 요구사항을 정리한다.

핵심은 다음 네 가지다.

1. 결과 항목마다 **원문 파일/페이지 근거 링크**를 제공한다.
2. 날짜와 내용의 관계를 단순 인접성으로 처리하지 않고 **문맥/양식 기반으로 귀속**한다.
3. **한 페이지 1날짜 + 다음 페이지 이어짐** 같은 문서 패턴을 이해한다.
4. 대규모 fine-tune에 기대기보다, **대응 엔진(구조 + 규칙 + 모델 + 검증)** 중심으로 문제를 푼다.

### 1.3 현재 개발 위치
현재 엔진은 다음 상태까지 와 있다.

- 기본 처리 파이프라인은 runtime 기준으로 안정성이 높아짐
- 날짜/병원명/사건 묶기 후처리 품질이 개선됨
- 그러나 제품화 관점에서는 여전히 tail-risk 케이스 제거가 우선
- 다음 설계/개발은 “평균 품질 향상”보다 **저점 제거와 검증 가능성 강화**가 우선

## 2. 문제 정의

### 2.1 현재 시장/문제 맥락
비정형 의료문서는 단순 OCR만으로는 실무 활용이 어렵다.

이유:
- 날짜가 앞 내용/뒤 내용을 수식하는 방식이 문서마다 다름
- 같은 병원이 여러 이름으로 등장할 수 있음
- 한 페이지에 날짜가 여러 개 있거나, 반대로 한 페이지 전체가 한 날짜일 수 있음
- 다음 페이지로 같은 날짜의 내용이 이어지는 양식이 존재함
- 결과가 맞아 보여도 사용자는 “어느 파일 몇 페이지냐?”를 반드시 확인하고 싶어함

즉, 이 문제는 “텍스트 추출” 문제가 아니라  
**구조 이해 + 날짜 귀속 + 근거 링크 + 낮은 꼬리 리스크 제거** 문제다.

### 2.2 사용자 관점 문제
사용자는 다음과 같은 불편을 겪는다.

- 결과가 맞는지 검증하기 어렵다
- 날짜와 사건의 연결이 어색할 수 있다
- 같은 병원이 여러 곳처럼 보일 수 있다
- 한 날짜 아래 묶여야 할 내용이 따로 떨어지거나, 반대로 섞일 수 있다
- 긴 문서에서 원문 근거를 다시 찾는 시간이 많이 든다

---

## 3. 제품 목표

### 3.1 핵심 목표
1. 의료문서를 **검증 가능한 타임라인**으로 바꾼다.
2. 각 타임라인 항목을 **원문 파일/페이지와 연결**한다.
3. 날짜와 내용의 관계를 문맥/양식 단위로 더 정확히 판단한다.
4. 애매한 케이스를 무리하게 자동 처리하지 않고 **불확실성 게이트**로 구분한다.
5. Starter / Pro 구조를 통해 **기본 엔진**과 **상위 검증층**을 분리한다.

### 3.2 성공 기준
다음 상태에 도달하면 v1 성공으로 본다.

- 주요 사건/날짜/기관 항목이 원문 링크와 함께 제공된다
- 날짜 귀속 오류가 tail-risk 케이스에서 감소한다
- single-date page / continuation 양식에서 오귀속이 줄어든다
- 애매한 케이스를 Pro 검증층으로 올릴 기준이 정해진다
- 제품이 “그럴듯한 요약기”가 아니라 **검증 가능한 엔진**으로 인식된다

---

## 4. 비목표(Non-Goals)

이 문서의 범위에 포함되지 않는 것:

- 모든 의료문서 양식을 100% 자동 이해하는 것
- 완전한 사람 수준 직관 복제
- 모든 애매한 사례를 규칙으로 하드코딩하는 것
- 대규모 fine-tune 모델 구축 자체
- 결제 시스템 세부 구현
- 베타/정식 출시 운영 정책 전체

---

## 5. 주요 사용자

### 5.1 1차 사용자
- 의료/보험/손해사정/법률 보조 문서를 다루는 실무자
- 원문 근거 검증이 필요한 사용자
- 긴 의료문서를 날짜 기준으로 정리하고 싶은 사용자

### 5.2 2차 사용자
- 문서 내용을 빠르게 훑고 핵심 사건을 이해하려는 사용자
- 결과에 대해 추가 질문을 하고 싶은 상위 플랜 사용자

## 6. 제품 구조 개요

### 6.1 Starter
**개념:** 검증 가능한 기본 엔진

**제공 내용**
- OCR + 좌표 기반 구조화
- 파일/페이지 출처 링크
- 기본 사건/날짜/기관 정리
- 타임라인 출력
- 애매한 항목 표시

**가치**
- 저비용
- 기본 품질 일관성
- 근거 기반 결과
- 실무 검토에 적합

### 6.2 Pro
**개념:** 애매한 부분만 상위 검증층으로 보완

**제공 내용**
- Starter 결과 포함
- 불확실성 게이트에 걸린 페이지/블록만 vision LLM 교차검증
- 설명형 질문 기능
- 근거 대조/보완 답변
- 더 높은 신뢰도

**원칙**
Pro는 전체 문서를 vision으로 처리하는 것이 아니라  
**선택적 vision**을 적용한다.

## 7. 핵심 기능 요구사항

### 7.1 Source Graph 구축
문서를 텍스트 덩어리가 아니라 구조 그래프로 저장해야 한다.

**최소 단위**
- 파일
- 페이지
- 블록
- 라인
- 토큰
- 날짜 후보
- 기관 후보
- 섹션 후보
- 사건 후보

**최소 속성**
- 텍스트
- 페이지 번호
- bbox / span
- confidence
- source_file_id / source_file_name
- 역할 추정(label / title / paragraph / table / metadata 등)

### 7.2 출처 링크 기능
모든 최종 타임라인 항목은 최소한 다음과 연결돼야 한다.

- source file
- page number
- block/span anchor
- 원문 미리보기 또는 하이라이트 가능한 참조 정보

**UX 요구**
- 결과 항목 클릭 시 원문 페이지로 이동 가능해야 한다
- Pro에서는 “이 판단의 근거는 어디인가?”에 대해 페이지 링크를 제공할 수 있어야 한다

### 7.3 날짜 귀속 엔진 (Temporal Binding Engine)
날짜와 내용의 관계를 다음 유형으로 구분해야 한다.

- forward binding
- backward binding
- mixed binding
- range binding
- carry-forward

**입력 신호**
- 날짜 후보 위치
- 날짜 주변 라벨
- 인접 블록 구조
- 페이지 내 날짜 개수
- 문서 family
- 다음 페이지 구조 변화
- 병원/섹션/표 구조 정보

**출력**
- 날짜별 사건 귀속 결과
- 귀속 방식 유형
- 근거 블록 링크
- 불확실성 점수 또는 상태

### 7.4 Single-Date Page / Continuation 처리
한 페이지가 하나의 날짜를 대표하고, 다음 페이지로 내용이 이어지는 양식을 처리해야 한다.

**감지 신호**
- 페이지 전체에서 유효 날짜가 1개
- 다음 페이지에 새 날짜 없음
- 같은 병원/양식 유지
- 문단/섹션 흐름 연속
- 새 헤더/새 문서 시작 부재

**출력**
- current page date carry-forward 여부
- carry-forward 종료 조건
- continuation confidence

### 7.5 기관명 정규화 / 사건 묶기
최소 수준의 다음 기능은 유지돼야 한다.

- 병원명 canonicalization
- alias collapse
- same-day event grouping
- 사건 분리/병합 일관성 유지

### 7.6 불확실성 게이트
시스템은 스스로 “여긴 애매하다”를 판정할 수 있어야 한다.

**게이트 입력 신호**
1. OCR/품질 신호
2. 구조/귀속 충돌 신호
3. 근거 부족 신호
4. 자기검증/대안해석 신호

**원칙**
- 한 신호만으로 판단하지 않는다
- 다중 신호 합산으로 escalation 여부를 정한다
- 자기검증은 보조 신호이지, 단독 근거가 아니다

**출력**
- `safe`
- `watch`
- `escalate_to_vision`

### 7.7 Selective Vision Cross-Check (Pro)
Vision LLM은 전체 문서에 무조건 쓰지 않는다.

**호출 조건**
- 불확실성 게이트가 임계값 초과
- 애매한 페이지/블록/표만 지정 가능
- 전체 문서가 아니라 pinpoint escalation 가능

### 7.8 설명형 질문 기능 (Pro)
사용자가 특정 결과를 더 구체적으로 검증할 수 있어야 한다.

**예시 질문**
- 이 날짜는 어느 파일 몇 페이지 근거인가?
- 왜 이 병원명으로 정규화됐는가?
- 이 사건이 왜 이 날짜 아래 묶였는가?
- 이 페이지는 이전 페이지와 같은 날짜가 이어지는가?

---

## 8. 품질 원칙

### 8.1 핵심 원칙
평균 품질 향상보다 **저점 제거**가 우선이다.

### 8.2 제품화 전 최소 기준
- 치명적 runtime 실패 거의 없음
- 회귀 케이스 0 또는 통제 가능 수준
- 명확한 tail-risk 케이스 거의 없음
- 결과 근거 링크 제공 가능
- 애매한 케이스의 escalation 경로 존재

---

## 9. 기술 원칙

### 9.1 철학
우리가 만드는 것은 “새로운 거대모델”이 아니라 다음의 조합이다.

- OCR/텍스트 추출
- 구조 그래프(Source Graph)
- 날짜 귀속 엔진
- 기관/사건 정규화
- 출처 링크
- 불확실성 게이트
- 선택적 vision 교차검증
- 평가/회귀 관리

### 9.2 금지할 방향
- 규칙 폭증
- 양식마다 if문 누적
- AI 자기판단만 신뢰
- 평균 개선만 보고 제품화 선언
- cost-aware design 없는 full vision

## 10. 성공 지표(초기)

### 기능 지표
- 결과 항목 중 출처 링크 부착 비율
- 날짜 귀속 성공률(내부 평가셋 기준)
- single-date continuation 오귀속 감소
- ambiguity gate escalation precision

### 품질 지표
- low-end risk present 케이스 수 감소
- tail-risk 케이스 수 감소
- regression count 유지/감소

### 운영 지표
- Starter 평균 처리 비용
- Pro selective vision escalation 비율
- Pro escalation 후 실제 개선 비율

## 13. 최종 정의

이 제품은 단순 OCR 앱이 아니고,  
그렇다고 새로운 foundation model을 발명하는 프로젝트도 아니다.

가장 정확한 정의는 다음과 같다.

> **비정형 의료문서를 검증 가능한 시간축 정보로 바꾸는 대응 엔진**
>
> 구성요소:
> - 구조 이해
> - 날짜 귀속
> - 파일/페이지 근거 링크
> - 저점 제거 중심 품질관리
> - 선택적 vision 검증

## 14. 한 줄 결론

우리가 만들어야 하는 것은  
**사람처럼 모든 걸 “느끼는” AI가 아니라,  
사람이 믿고 검증할 수 있는 구조와 근거를 제공하는 엔진**이다.
