# 비정형 의료문서 대응 엔진 설계 메모
_논의 종합 정리 / 개발 방향 문서_

## 문서 목적

이 문서는 지금까지 논의한 핵심 문제의식과 판단을 한 번에 정리한 개발 메모다.  
목표는 다음과 같다.

1. 우리가 실제로 풀어야 하는 문제를 다시 명확히 정의한다.
2. 어떤 방향이 현실적이고, 어떤 방향이 과장되거나 비효율적인지 구분한다.
3. 앞으로 엔진/모듈/파이프라인을 어떤 우선순위로 구축할지 합의 가능한 기준을 만든다.
4. 제품 구조(Starter / Pro)와 신뢰성 확보 방식(출처 링크, 불확실성 게이트, selective vision)을 정리한다.

이 문서는 “좋은 아이디어 모음”이 아니라, **현실적인 구현 방향과 판단 기준을 정리한 작업 문서**다.

---

# 1. 출발점: 네 가지 핵심 문제의식

## 1-1. 페이지 링크 / 출처 추적 문제

### 문제의식
정리된 결과가 좋아 보여도, 결국 사용자는 이렇게 묻게 된다.

- 이 내용은 **어느 파일**에서 나온 것인가?
- **몇 페이지**에 있는가?
- 원문에서 바로 확인할 수 있는가?

특히 의료/보험 문서는 “잘 요약했다”보다 **원문 근거로 점프할 수 있는가**가 훨씬 중요하다.

### 현재 판단
이건 선택 기능이 아니라 **신뢰 기능**이다.  
베타/유료화 이전에 반드시 구조를 잡아야 한다.

### 필요한 최소 정보
결과 객체마다 아래 정보가 붙어야 한다.

- `source_file_id`
- `source_file_name`
- `page_number`
- `bbox` 또는 `span`
- 필요하면 `anchor_text`

### 결론
**출처 링크는 P0 우선순위**다.

---

## 1-2. 날짜-내용 귀속 문제

### 문제의식
비정형 의료문서에서는 날짜와 내용의 관계가 단순하지 않다.

예시 A:
- `A날짜 ~ B내용 ~ C날짜 ~ D내용`
- 보통 B는 A에, D는 C에 속한다.

예시 B:
- `E내용 ~ F날짜 ~ G내용 ~ H내용 ~ I날짜 ~ J내용`
- 특정 양식에서는 E가 F를, H가 I를 수식할 수도 있다.

즉, 날짜가 항상 “뒤 내용을 먹는 anchor”가 아니다.  
경우에 따라서는 **앞 내용을 수식**하거나, **양쪽 일부를 먹거나**, **구간을 선언**할 수도 있다.

### 현재 판단
이건 단순 OCR 문제가 아니라 **chronology 엔진의 핵심 로직**이다.  
“가까운 날짜에 붙인다”로는 해결되지 않는다.

### 핵심 정의
날짜 귀속에는 최소한 다음 유형을 구분해야 한다.

- `forward binding`
- `backward binding`
- `mixed binding`
- `range binding`
- `carry-forward`

### 결론
이건 후처리의 작은 예외가 아니라, **별도 모듈로 설계해야 하는 중심 문제**다.

---

## 1-3. 1페이지 1날짜 양식 + 다음 페이지 이어짐 문제

### 문제의식
대부분의 의료문서는 한 페이지에 날짜별 내용이 연속해서 나오지만, 예외적으로:

- 페이지 전체가 하나의 날짜를 의미하고
- 다음 페이지에도 같은 날짜 내용이 이어지는

양식이 존재한다.

이 경우,
- 페이지 상단/하단 어딘가에 날짜가 한 번만 표기되고
- 다음 페이지에는 새 날짜가 안 나오는데
- 실제로는 앞 페이지 날짜가 계속 유효할 수 있다.

### 현재 판단
이건 충분히 구현 가능한 문제다.  
다만 단순 텍스트만 보면 약할 수 있고, **문서 패턴/양식 이해**가 필요하다.

### 필요한 판단
- 이 페이지가 `single-date page`인지
- 다음 페이지가 `continuation`인지
- 언제 carry-forward를 멈춰야 하는지

### 결론
이 역시 별도 규칙/모듈로 다뤄야 한다.  
특히 페이지 패턴 이해가 필요하다.

---

## 1-4. fine-tune보다 “대응 엔진” 철학

### 문제의식
대량 케이스 학습으로 거대한 전용 모델을 만들기보다,  
비정형문서에 **대응할 수 있는 엔진/로직/모듈**을 구축하고 싶다는 철학.

### 현재 판단
이 방향은 과한 게 아니라 오히려 **더 현실적**이다.

왜냐하면 이 문제는:
- 양식이 다양하고
- 예외가 많고
- 한 번 틀리면 신뢰를 잃는 영역이며
- 순수 end-to-end 모델 하나로 끝나지 않기 때문이다.

### 단, 주의점
이 철학이 규칙 만능주의로 가면 망한다.

좋은 방향:
- 공통 구조화 가능한 것만 엔진화
- 문서 family 단위 패턴 처리
- 불확실하면 보류/에스컬레이션
- 평가셋으로 회귀 검증

나쁜 방향:
- 문제 나올 때마다 if문 추가
- 문서 하나마다 규칙 하나씩 누적
- 사람이 보던 감각을 규칙으로 모두 복제하려고 시도

### 결론
**“대응 엔진” 철학은 맞다.**  
다만 **만능 규칙 덩어리**가 아니라,  
`모델 + 구조 + 규칙 + 평가`의 균형 구조여야 한다.

---

# 2. 핵심 판단: 사람의 직관을 그대로 복제할 수 있는가?

## 짧은 답
완전 복제는 어렵다.  
하지만 **기능적으로 흉내 내는 것**은 충분히 가능하다.

## 왜 그런가
사람이 문서를 잘 읽는 이유는:
- 제목/본문/잡음을 감으로 구분하고
- 날짜가 앞을 먹는지 뒤를 먹는지 감으로 알고
- 양식의 반복성을 빨리 느끼고
- 애매하면 애매하다고 판단하기 때문이다.

이걸 시스템화하려면 “직관”을 하나로 복제하는 게 아니라,  
직관을 구성하는 기능을 모듈로 쪼개야 한다.

### 필요한 기능 분해
- 양식 분류
- 레이아웃 분할
- 날짜 귀속
- 기관명 정규화
- 사건 묶기
- 출처 링크
- 불확실성 판단

즉, 사람의 시각을 복제한다기보다  
**사람의 판단 기능을 분해해서 재조합**하는 게 맞다.

---

# 3. OCR 텍스트 + 좌표만으로 어디까지 가능한가

## 결론
**출발은 가능하다.**  
그리고 1차 시스템은 충분히 만들 수 있다.

하지만 2번/3번 문제를 강하게 풀려면  
결국 **이미지/레이아웃 보조 신호가 필요해질 가능성**이 높다.

## OCR 텍스트 + 좌표만으로 잘 되는 것
- 파일/페이지/좌표 기반 출처 링크
- 같은 줄/박스/표셀 기반 귀속
- 날짜 anchor와 내용 묶기
- 상단/하단 반복 패턴 탐지
- 단순 표/문단 분리
- 날짜가 앞/뒤 내용을 수식하는 방향성 추정

## OCR 텍스트 + 좌표만으로 약한 것
- 박스/음영/시각적 구분선
- 양식의 시각적 관성
- 페이지 이어짐을 시각 패턴으로 감지
- 표처럼 보이지만 텍스트만 보면 표가 아닌 경우

## 실전 결론
현실적인 전략은:

1. **텍스트 + 좌표 기반 1차 구현**
2. hard case에만 **이미지/vision 보조**
3. 완전 전면 vision은 비용과 속도 때문에 피함

# 4. 우리가 지향하는 엔진 구조

## 4-1. Source Graph
문서를 “텍스트 덩어리”가 아니라 “구조 그래프”로 본다.

### 단위
- 파일
- 페이지
- 블록
- 라인
- 토큰
- 표셀
- 날짜 후보
- 기관 후보
- 섹션 후보

### 각 노드가 가지는 값
- 텍스트
- 페이지 번호
- bbox/poly
- confidence
- 주변 라벨
- 역할 추정
- source link

### 목적
이 그래프가 있어야 나중에:
- 날짜 귀속
- 병원 정규화
- 사건 묶기
- 링크 추적

이 모두가 가능해진다.

---

## 4-2. Temporal Binding Engine
이게 날짜 귀속 엔진이다.

### 핵심 기능
- forward binding
- backward binding
- mixed binding
- range binding
- carry-forward
- single-date-page continuation

### 핵심 입력
- 날짜 후보 위치
- 주변 블록/라인
- 라벨 cue
- 페이지 내 날짜 수
- 다음 페이지의 구조 변화
- 문서 family 정보

### 목적
“이 내용은 어느 날짜에 속하는가?”를 판단한다.

---

## 4-3. Template / Family Detector
문서를 개별 문서 단위가 아니라 **양식 family** 단위로 본다.

### 예시 family
- 날짜-내용 연속형
- 페이지당 단일날짜형
- 검사결과표형
- 입퇴원 서식형
- 판독지형
- 진료기록표형

### 목적
문서마다 규칙을 만드는 대신,  
family별 정책을 적용한다.

---

## 4-4. Evidence Linker
최종 결과를 다시 원문과 연결한다.

### 예시
- 이 사건은 파일 A 3페이지 블록 5에서 추출됨
- 이 날짜는 파일 B 2페이지 표 셀에서 옴

### 목적
- 사람 검증 가능
- 질문 기능 가능
- 신뢰도 상승
- 오류 시 빠른 디버깅 가능

---

## 4-5. Low-End Risk Gate
애매하거나 위험한 케이스를 자동 감지한다.

### 예시 신호
- 날짜 귀속 충돌
- 병원명 후보 충돌
- 근거 약함
- carry-forward 불확실
- mixed binding 충돌
- OCR 품질 낮음

### 목적
평균 품질보다 **저점 제거**를 먼저 한다.

---

# 5. Starter / Pro 제품 구조 제안

## 5-1. Starter
### 개념
**검증 가능한 기본 엔진**

### 제공 가치
- OCR + 좌표 기반 구조화
- 파일/페이지 출처 링크
- 기본 타임라인/정리
- 애매한 부분 표시
- 일관된 기본 결과

### 강점
- 비용 예측 가능
- 속도/일관성 좋음
- 신뢰 가능한 기본 기능

### 주의점
Starter에서 너무 많은 해석을 약속하면 안 된다.

---

## 5-2. Pro
### 개념
**애매한 부분만 상위 검증층으로 보완**

### 제공 가치
- Starter 결과 포함
- ambiguity pages / ambiguous blocks만 vision cross-check
- 질문 기반 검증
- 설명형 QA
- 보다 높은 신뢰도

### 중요한 원칙
Pro는 “전면 vision”이 아니라  
**선택적 vision**이어야 한다.

---

# 6. 불확실성 게이트 설계 원칙

## 핵심 질문
“어떤 문서/페이지/블록이 vision 검증이 필요한가?”

## 내 결론
한 신호로 판단하면 안 된다.  
반드시 **다중 신호 게이트**로 가야 한다.

---

## 6-1. 1차 게이트: 구조/품질 신호
예:
- OCR 품질 낮음
- 날짜 귀속 충돌
- 병원명 충돌
- 근거 링크 약함
- single-date carry-forward 불확실
- competing interpretation 존재

이건 엔진 내부에서 이미 감지 가능한 신호다.

---

## 6-2. 2차 게이트: 자기검증 질문
예:
- 이 날짜는 진료일인가 작성일인가?
- 이 병원명 정규화는 근거가 충분한가?
- 이 페이지는 이전 페이지와 같은 날짜가 이어지는가?
- source evidence가 충분한가?
- 다른 해석이 더 그럴듯한가?

### 주의점
자가검증은 유용하지만, **혼자서는 불충분**하다.  
LLM self-reflection은 흔들릴 수 있다.

즉, 자기검증은 **보조 게이트**다.

---

## 6-3. 3차 게이트: disagreement / escalation
다음이 동시에 일어나면 escalation:
- 규칙 엔진 판단
- 자기검증 결과
- 기존 구조 해석

이 셋이 충돌할 때.

그때만:
- 전체 문서가 아니라
- **애매한 페이지**
- **애매한 블록**
에만 vision LLM을 쓴다.

---

## 6-4. 신뢰도 확보 방법
게이트도 검증 대상이어야 한다.

### 측정해야 할 것
- 놓침(false negative)
- 과잉 escalation(false positive)
- escalation precision
- escalation 후 실제 개선율

즉, “vision을 많이 썼다”가 아니라
**필요한 곳에만 잘 썼는가**를 본다.

---

# 7. 네 4가지 문제의식에 대한 종합 판단

## 7-1. 페이지 링크
필수.  
제품 신뢰의 핵심.

## 7-2. 날짜 귀속
핵심 엔진 문제.  
별도 모듈 필요.

## 7-3. 한 페이지 패턴 / 다음 페이지 이어짐
충분히 구현 가능.  
문서 family + carry-forward 모듈로 다뤄야 함.

## 7-4. fine-tune보다 대응 엔진
맞는 철학.  
다만 규칙 만능주의는 피해야 함.

## P1
- 날짜 귀속 엔진
- single-date page continuation 처리
- 불확실성 게이트 v1

## P2
- selective vision cross-check
- 설명형 질문 기능
- productization / beta 판단

# 9. 비현실적인 방향 vs 현실적인 방향

## 비현실적인 방향
- giant fine-tune만으로 해결하려 하기
- 문서 하나마다 규칙 추가
- 전면 vision LLM 호출
- AI 자기판단만 믿기
- 평균 품질만 보고 제품화 결정

## 현실적인 방향
- Source Graph 구축
- 날짜 귀속 엔진 분리
- family 단위 패턴 처리
- evidence link 필수화
- low-end risk 중심 품질판단
- ambiguity만 selective vision

# 10. 최종 철학 정리

## 우리가 만들고 있는 것은 무엇인가
이 프로젝트는 단순 OCR 앱이 아니다.  
그렇다고 “새로운 AI 모델 발명 프로젝트”도 아니다.

가장 정확한 정의는:

> **비정형 의료문서에 대응하는 구조 이해 + 귀속 판단 + 출처 링크 + 저점 제거 중심의 실무형 엔진**

이다.

## 핵심 원칙
1. **근거 없는 자동화는 하지 않는다**
2. **애매하면 보류/에스컬레이션한다**
3. **평균보다 저점 제거를 우선한다**
4. **규칙 만능주의가 아니라, 구조+규칙+모델+평가의 균형으로 간다**
5. **Starter는 검증 가능한 기본 엔진, Pro는 선택적 상위 검증층으로 설계한다**
