# 의료문서 분석 앱 전환 정리 문서

작성일: 2026-03-23  
목적: 손해사정보고서 자동작성 중심 규칙을, 설계사 실사용 중심의 의료문서 분석 앱 방향으로 재구성하고, 기존 문서 수정 및 신규 문서 추가 계획까지 한 번에 정리한다.

---

## 1. 한 줄 결론

현재 앱은 **의료문서를 읽고 의료 이벤트를 구조화하는 엔진** 쪽은 상당히 진척되어 있다.  
하지만 **설계사가 고객 보험금 청구 건을 분석하는 상품**으로서의 포장, 화면, 가입, 결제, Starter/Pro 구분은 아직 명시적으로 재정렬되지 않았다.

즉, 지금 필요한 것은 엔진을 새로 만드는 것이 아니라,

1. 기존 손해사정 보고 규칙을 **앱용 분석 출력 체계**로 재정의하고
2. 설계사 단일 타깃 기준으로 **제품 문서와 기능 우선순위**를 다시 정렬하며
3. Starter / Pro / 가입 / 결제 / 질문 기능을 **제품 문서에 먼저 반영**하는 것이다.

---

## 2. 첨부 문서에서 건져야 할 핵심

### 2-1. 고지의무 문서에서 가져갈 핵심

핵심은 단순 보고서 작성이 아니다.  
**보험가입일을 기준으로 전체 의무기록을 기간별로 나누고, 고지의무 해당 가능 진료를 자동 태깅하는 분석 엔진**이 필요하다는 점이다.

문서에서 반복되는 핵심 규칙:

- 병원 2~3곳 이상 기록도 한 번에 통합 분석
- 보험가입일 기준으로 가입 전/후 전체 타임라인 재구성
- 3개월 / 2년 / 5년 같은 고지 구간을 상품별로 가변 적용
- 고지의무 후보 진료 자동 분류
- 확장형 보고서 + 결재용 요약본 2단 출력
- 실제 사실만 쓰고, 예시는 하드코딩하지 않음

이 문서군은 앱의 핵심 기능을 사실상 이렇게 정의한다.

> “OCR 앱”이 아니라, **보험가입일 기준 전체 의료기록 리스크 분석기**

---

### 2-2. 손해사정 보고서 프롬프트에서 가져갈 핵심

보고서 형식의 본질은 이미 꽤 정리돼 있다.

핵심 항목:

- 내원일시
- 내원경위
- 진단병명(KCD-10)
- 검사결과(원어 + 한글 병기)
- 수술 후 조직검사 결과(암일 때)
- 치료내용
- 통원기간 / 입원기간 분리
- 과거병력
- 의사소견
- 일자별 경과표

특히 실무적으로 중요한 규칙:

- **통원과 입원을 반드시 분리**
- **당일 입원은 1일 입원으로 처리**
- 검사명과 결과는 붙여 쓰고, 가능하면 원어와 한글을 같이 표시
- 암은 보고일, 병기, 조직소견까지 별도 구조화
- 동일 환자의 여러 문서는 연대순 경과표로 재구성

이 형식은 그대로 버릴 것이 아니라, **앱의 출력 레이어 중 하나로 흡수**해야 한다.

---

### 2-3. 질환별 검사결과 규칙 문서에서 가져갈 핵심

이 문서는 단순한 텍스트 템플릿이 아니라, 앱이 가져야 할 **질환군별 출력 규칙 엔진**의 씨앗이다.

질환군별 핵심:

- 협심증: CT / MRI / CT-Angio / MRA / 관상동맥조영술, 협착률, TIMI
- 급성심근경색: 위 항목 + Troponin / CK-MB / EKG / PCI / stent / TIMI
- 부정맥: EKG 리듬, Holter, HR 변화
- 뇌혈관질환: Brain CT / CTA / MRI / MRA / Angio, 병변 위치와 범위
- 암: 검사일 / 보고일 / 진단일 / 확진일 / 판독일 / TNM / 원발·전이
- 기타 질환: 단순 수치보다 정밀 영상 우선

이건 곧 다음 의미다.

> 앱은 단순 요약이 아니라, **질환군별로 중요한 검사만 골라 정리하는 도메인 출력 엔진**이 되어야 한다.

---

### 2-4. 최종 요약 규칙에서 가져갈 핵심

결재용 최종 요약본은 여전히 필요하다.  
다만 이건 이제 앱의 “최종 산출물 중 하나”이지, 앱 전체 정체성 그 자체는 아니다.

유지해야 할 역할:

- 핵심 내용만 압축
- 날짜, 병기, 검사결과, 입퇴원, 치료요약 중심
- KCD-10 + 영문 병명 병기
- 주치의 소견 한 줄 요약

즉,

- **Starter**에서는 기본 요약 결과의 마지막 섹션으로 제공
- **Pro**에서는 심층 분석 후 “결재용 요약본” 또는 “공유용 요약본”으로 파생 제공

하는 구조가 맞다.

---

## 3. 현재 앱 개발상황을 제품 언어로 다시 설명하면

### 이미 상당히 구현된 것

1. 여러 문서에서 날짜/병원/이벤트를 뽑아 구조화하는 핵심 엔진
2. 애매한 근거나 약한 묶음을 “검토 필요”로 표시하는 안전장치
3. 조사자용 보고/서술 출력에서 review signal을 전달하는 출력 품질 강화
4. 고지의무 분석으로 확장할 수 있는 가입일 기준 구조

### 아직 부족한 것

1. 설계사 단일 타깃 제품 정의
2. Starter / Pro 상품 문서와 실제 화면 흐름
3. 설계사 관점의 “한눈에 보는 사건 개요” 계층
4. 질문형 분석(Pro Q&A / 검색 / 재질문)
5. 회원가입 간소화, 소셜 로그인
6. 결제 / 구독 체계

즉, 현재는

- **엔진은 앞서감**
- **제품화는 뒤처짐**

상태다.

## 4. “손해사정 보고서 작성기”에서 “의료문서 분석 앱”으로 바뀔 때, 출력 구조를 어떻게 바꿔야 하나

핵심은 보고서를 없애는 게 아니다.  
보고서는 **최종 출력의 한 종류**로 낮추고, 앱의 중심 출력은 **분석 결과 화면**으로 바꿔야 한다.

### 새 출력 구조 제안

#### A. 공통 코어 분석 결과
모든 상품 공통

- 문서 통합 현황
- 병원별/기간별 문서 개수
- 전체 의료 이벤트 타임라인
- 주요 질환군 자동 분류
  - 암
  - 심장
  - 뇌혈관
  - 수술
  - 입원
  - 추가검사 필요 소견
  - 의심 / 확정 진단
- 보험가입일 기준 고지 구간 분류
- 근거 문서/페이지 링크
- confidence / review-needed 표시

#### B. Starter 결과 화면
설계사 현장 실무용 1차 분석

- 전체 사건 개요
- 주요 의료 이벤트 시계열
- 질환군별 개괄 요약
- 보험가입일 기준 고지의무 검토 개요
- 청구 전후 맥락 요약
- 변동 가능성 / 누락 가능성 / 정확도 주의 문구
- 의료적/보험적 최종 판단 금지 고지
- 공유/출력 가능한 기본 요약 리포트

#### C. Pro 결과 화면
심층 분석 및 재질문용

- Starter 전체 포함
- selective vision cross-check / 고급 교차검증
- 더 깊은 질환군별 검사 해석 정리
- 질문/검색/재질문 기능
- “왜 이렇게 분류됐는지” 설명
- 문서 원문 근거로 drill-down
- 고지의무 후보 진료 상세 검토
- investigator/전문가 수준에 가까운 상세 리포트 / 의견 초안

#### D. 리포트 출력
앱 결과를 문서로 내보낼 때

1. 기본 분석 리포트(Starter)
2. 심층 분석 리포트(Pro)
3. 결재용 요약본
4. 필요 시 손해사정 보고서 스타일 출력

즉, 앞으로는 **앱이 중심**이고, **보고서는 파생물**이어야 한다.

---

## 5. 새 보고서 양식 제안

기존 “손해사정 보고서” 양식은 유지하되, 앱 기준으로는 아래 3계층으로 나누는 게 맞다.

### 5-1. Starter 기본 분석 리포트

#### 1. 사건 기본정보
- 피보험자/청구 건 식별정보
- 보험가입일
- 적용 고지 기준(예: 3.2.5 / 3.1.5 / 사용자 입력 기준)
- 업로드 문서 개요

#### 2. 전체 의료 이벤트 타임라인
- 날짜
- 병원
- 이벤트 유형(외래/입원/수술/검사/병리/응급 등)
- 핵심 내용
- 근거 링크

#### 3. 핵심 질환군 개요
- 암
- 심장
- 뇌혈관
- 수술
- 입원
- 주요 만성질환
- 각 항목별 “있음 / 없음 / 검토 필요”

#### 4. 고지의무 검토 개요
- 3개월 이내
- 1년 또는 2년 이내
- 5년 이내
- 상품 기준 미기재 시 기본규정 적용 안내
- 후보 진료 목록 요약

#### 5. 주의사항
- 의료적 최종 진단 아님
- 보험금 지급 판단 아님
- 원문 누락/스캔 품질/문서 편차에 따라 달라질 수 있음

#### 6. 기본 요약본
- 설계사가 고객/내부 검토용으로 바로 보는 요약

---

### 5-2. Pro 심층 분석 리포트

#### 1. Starter 전체 포함

#### 2. 질환군별 정밀 분석
- 질환군별 핵심 검사 자동 선별
- 원어 + 한글 병기
- 날짜 체계 정렬
- 모순/불일치/추가확인 포인트

#### 3. 고지의무 상세 검토
- 상품 기준별 기간 계산 근거
- 후보 진료를 왜 해당 후보로 분류했는지
- 의심 / 확정 / 수술 / 입원 / 추가검사 필요 소견 근거

#### 4. 심층 질문 기능 결과
- 사용자가 “이 기록이 3개월 이내 고지대상인가?” 같이 질문 가능
- 답변은 원문 근거와 함께 제시

#### 5. 고급 요약 리포트
- 설계사 내부 검토용
- 필요 시 전문가/손해사정 연결 전 브리핑용

---

### 5-3. 문서 출력용 최종 요약본

이건 기존 결재용 요약 양식을 유지한다.

- 내원일시
- 내원경위
- 진단병명
- 검사결과
- 수술 후 조직검사 결과(암일 때)
- 치료내용
- 통원기간
- 입원기간
- 과거병력
- 의사소견

다만 앱에서는 이걸 “최종 결과 중 하나”로 제공한다.

## 6. 새 방향 5개를 현재 계획에 대입한 판단

### 6-1. 설계사 단일 타깃
판정: **새로 재정의 필요**

첨부된 codex 정리 기준, 기존 로컬 계획 문서에는 consumer / investigator 이원 구조가 있었고, 진짜 새 방향은 “설계사 단일 타깃 재정의”다. 즉 이 항목은 기존 문서를 수정해야 한다.

### 6-2. Starter
판정: **기존 개념은 있었고, 지금 더 실무형으로 재정의 필요**

Starter 자체는 기존 계획에 있었지만, 지금 필요한 Starter는

- 설계사가 현장에서 바로 보는 1차 분석 도구
- 고지의무 개요 포함
- 질환군 개괄 포함
- 판단 금지 문구 포함

형태로 더 구체화해야 한다.

### 6-3. Pro
판정: **기존 개념은 있었고, 기능 정의를 더 구체화해야 함**

Pro는 원래도 selective vision, 질문 기능, 더 깊은 분석 구조가 있었다는 codex 정리가 있다. 다만 지금은 설계사 단일 타깃 기준으로 “언제 무엇을 추가로 제공하는지”를 다시 써야 한다.

### 6-4. 회원가입 간소화 / 소셜 로그인
판정: **기획 흔적은 있었고, 구체 정책은 다시 써야 함**

현재는 이름/이메일 중심 간편 가입, 구글/카카오 연동 등 실제 정책이 제품 문서에 명확히 정리돼야 한다.

### 6-5. 결제 / 구독
판정: **기획은 있었고, 상품 구조 기준으로 다시 써야 함**

기존 billing/plan 구조는 있었던 것으로 보이지만, 이제는

- Starter 무료 / 체험 / 유료 여부
- Pro 업셀 구조
- 월 구독 / 건당 결제 / 크레딧

같은 실제 상품 모델을 다시 정해야 한다.

## 7. 기존 문서 수정 계획

초기 문서는 이 세션에서 직접 볼 수 없으므로, 아래는 **수정 대상과 수정 방향을 정리한 계획**이다.

### 최우선 수정 문서

#### 1. MASTER_PRD 계열
수정 방향:
- 타깃을 “일반사용자 + 조사자”에서 **설계사 단일 핵심 타깃**으로 재정의
- 제품 한 줄 정의를 “의료문서 분석 + 보험금 청구 사전검토 도구”로 수정
- 주요 사용자 시나리오를 설계사 기준으로 다시 작성

#### 2. AUTH / ROLES / PLANS / BILLING 문서
수정 방향:
- consumer / investigator 역할 구조 삭제 또는 후순위화
- 설계사 단일 role 기본값으로 수정
- 이름/이메일 간편 가입 정의
- Google / Kakao / 추가 OAuth 정책 명시
- Starter / Pro 요금제 구조 반영

#### 3. 의료 비정형 엔진 PRD / 전략 메모
수정 방향:
- 엔진 목적을 “보고서 자동작성”에서 “분석 앱 코어 엔진”으로 명확히 이동
- 출력 계층을
  - Starter 분석 결과
  - Pro 심층 분석
  - 문서 출력
  으로 분리
- 고지의무 분석을 코어 기능으로 승격

#### 4. LLM / Prompt 전략 문서
수정 방향:
- 기존 손해사정보고서 프롬프트들을
  - Core extraction
  - Disease-specific interpretation
  - Disclosure review
  - Final summary
  로 역할 분리

## 9. Codex가 수정할 때 지켜야 할 원칙

1. 엔진 문서를 새로 갈아엎지 말고, **설계사 단일 타깃 제품 방향** 위로 재정렬할 것
2. Starter / Pro / 로그인 / 결제는 “처음 추가”가 아니라, **기존 계획과 새 방향을 정합화**할 것
3. 기존 손해사정 보고 규칙은 버리지 말고, **앱 출력 포맷으로 흡수**할 것
4. “보고서 자동작성기”에서 “분석 앱”으로 옮기되, 문서 출력은 유지할 것
5. 의료적 판단 / 보험금 지급 판단을 앱이 단정하지 않도록 경고 체계를 명시할 것
6. 고지의무 분석은 상품별 기간이 가변적이므로 config 기반으로 설계할 것
7. Pro의 질문 기능은 반드시 **근거 링크 기반 설명형 답변**으로 정의할 것

## 11. Codex 전달용 지시문 초안

아래는 다음 세션에서 Codex가 기존 문서를 수정/추가할 때 바로 이해할 수 있도록 정리한 초안이다.

### 목적
현재 저장소의 엔진 중심 문서를, 설계사 단일 타깃 의료문서 분석 앱 방향으로 재정렬한다.

### 핵심 변경
- 일반사용자/조사자 이원 구조를 설계사 단일 핵심 타깃 구조로 재정의
- Starter / Pro를 손해사정 보고서 자동작성 기준이 아니라 의료문서 분석 앱 상품 기준으로 재정의
- 기존 손해사정 보고 규칙을 앱 출력 스키마 안으로 흡수
- 가입/로그인/소셜 연동/결제는 제품 문서에 명시적으로 반영

### Codex 작업 범위
1. 기존 PRD / 전략 / prompt / auth-billing 문서 수정
2. 새 제품 방향 문서 추가
3. 새 출력 스키마 문서 추가
4. Starter / Pro 정의 문서 추가
5. 가입/결제/질문기능 정책 문서 추가

### Codex 작업 금지
- 엔진 검증 기준선 재오픈
- 승인된 slice1 / slice2 재수정
- 실제 결제/로그인 코드 구현
- productization 코드 착수

### 최종 산출물 목표
- 설계사 단일 타깃 제품 정의 문서
- Starter / Pro 상품 정의 문서
- 앱용 분석 리포트 스키마 문서
- 고지의무 분석 엔진 규칙 문서
- 가입/로그인/결제 정책 문서
- 기존 문서 정합화

## 12. 최종 판단

지금 가장 중요한 건 기능을 더 붙이는 것이 아니다.  
**현재까지 만든 엔진을, 설계사가 실제로 쓰는 의료문서 분석 서비스 구조로 다시 번역하는 것**이다.

즉 다음 순서는 이렇게 가야 한다.

1. 제품 방향 문서 재정렬
2. Starter / Pro 정의 문서 확정
3. 앱 출력 스키마 재정의
4. 가입/결제/질문기능 정책 명시
5. 그 다음 구현 착수

이 순서를 거치지 않으면, 엔진은 좋아도 상품은 계속 흐려진다.
